Deployment

部署方式

Deployment Thesis控制面可以统一，执行位置不必统一。

这正是 ExecGov 当前能同时承接正式交付、公开体验和本地接入的关键原因。

ExecGov 的部署思路不是只有一种，而是根据 数据边界、客户规模和执行位置 做组合。 平台的目标不是强迫所有用户走同一条路径，而是用同一套治理逻辑覆盖不同落地形态。

对公开口径来说，当前更稳的路径是： 免费线、 本地脚本位轻量扩容、 共享 SaaS 正式团队版、 企业单租户 / 私有化交付， 再按需要叠加本地混合执行。

01

单租户交付模式

  • 适合企业客户项目交付
  • 需要独立访问地址
  • 需要和其他客户完全隔离
  • 一个客户一个租户，能力与数据边界清晰

02

平台治理模式

  • 平台方统一管理多个客户
  • 需要公告、模板、日志和台账总控
  • 需要跨租户治理但不打破隔离
  • public 作为平台治理域

03

免费入口模式

  • 适合产品体验入口和个人开发者免费线体验
  • 更强调上手速度和体验路径
  • 为后续更深沟通和正式交付提供前置入口

Path To Deployment

四层路径分别该怎么理解部署

当前层级更适合的部署理解当前重点下一步信号
免费线先用公开体验、登录 / 注册和个人入口理解平台先确认执行链路、产品定位和入口体验是否成立如果开始持续使用，就进入个人入口或继续看本地脚本位
本地脚本位增购个人空间轻量扩容，不改变整体部署形态解决个人持续接脚本的问题，不等于正式团队交付如果开始多人协作，就进入标准团队版评估
标准团队版优先共享 SaaS 正式租户入口适合标准能力、较轻协作和不需要深度前端定制的团队如果需要更强隔离、专属页面或正式验收，再转企业交付
企业交付单租户交付、私有化或更深部署控制面向内网、更强隔离、专属品牌或正式项目交付如果又依赖本地环境和内网资源，再叠加混合执行

04

本地混合执行模式

  • 适合文件和数据不能全部上云
  • 任务依赖本地环境或内网资源
  • 平台负责入口、规划、权限和审计
  • 本地环境负责执行，CLI / Agent 负责桥接
  • 客户本地脚本路径不要求固定盘符或固定目录，按当前机器真实路径登记即可

05

当前公开重点之外的路线

  • 当前公开重点不是设备、边缘或插件治理
  • Agent 常驻执行仍属于后续增强，不是现在的公开卖点
  • 先把真实项目里的本地执行和内网接入做好更重要
  • 更远路线等有真实需求再继续展开

Local Registration

客户本地路径和机器绑定，当前怎么落地

路径可以因人因机而异

  • 客户不需要把所有脚本统一放到某个固定盘符。
  • 同一个客户的不同脚本，可以分别登记不同绝对路径。
  • 如果客户只有一个租户，就只在自己的租户下登记本机路径即可。
  • 免费版当前默认对应 tenant_1000，但本地路径仍按用户自己的机器来定。

机器绑定必须收口

  • 同一个 Skill 当前不能同时绑多台机器。
  • 正式使用按“每台机器各自登记自己的脚本 / Skill”推进。
  • 平台派发本地作业时，会按登记时的机器标识路由到对应设备。
  • 这让控制面可以统一，但执行位置不必统一。

Shared SaaS

标准 SaaS 共享租户模式

当前怎么运行

从 tenant_1003+ 开始，标准 SaaS 客户会优先进入共享租户前端骨架。运行时按访问 Host 绑定 schema，所以仍然是租户隔离，不是把所有客户塞进同一个后台账号里混用。

适合什么场景

更适合标准能力、较轻团队协作和不需要深度前端定制的场景。如果客户需要更重的隔离、专属页面、专属品牌或更深部署控制，再继续走单租户交付或私有化。

  • 共享的是前端工程与正式发版节奏，不共享租户数据、账号和权限
  • 标准 SaaS 客户仍然按自己的租户空间使用能力、菜单、公告和结果
  • 当前共享租户前端已经建立，但按租户动态品牌化和更细部署参数还在继续收口

Upgrade Path

从免费入口到正式交付，当前更真实的升级路径

阶段当前状态更适合的部署理解下一步信号
01. 公开体验期先看公开体验页、登录页、注册页和公开文档这时重点不是一开始就谈私有化，而是先确认执行链路、产品定位和入口体验是否成立如果要反复使用，就进入个人入口或继续看本地脚本位 / 团队路径
02. 个人免费入口期用户已经进入免费入口，开始理解智能执行台、AI 执行和基础使用路径这时更接近产品体验与增长入口，不等于已经进入正式团队 / 企业交付如果只是个人持续接脚本，就先看本地脚本位；如果开始关心多人协作，再进入团队评估期
03. 个人扩容期已经在个人空间持续接脚本，但还没进入多人协作这时重点是个人空间扩容和本地接入，不需要先切到企业部署口径如果开始多人协作、权限控制和正式账号管理，再进入标准团队版
04. 标准团队期开始带着真实协作、真实权限问题和标准菜单需求来判断是否够用这时优先评估共享 SaaS 正式租户入口，而不是默认上来就单租户或私有化如果需要独立网址、专属页面和正式验收，就进入企业交付
05. 正式交付期客户要拿到自己的访问地址、自己的能力边界和正式交付包当前有两条更现实的路径：标准 SaaS 共享租户，或企业单租户交付。前者偏标准化，后者偏更强隔离与更深定制。如果只是标准协作与标准菜单，优先共享 SaaS；如果需要专属环境和更深定制，再走单租户交付
06. 本地扩展期任务开始依赖本地环境、内网资源或私有数据在单租户交付或平台治理之上叠加本地混合执行，由 CLI / Agent 承接桥接如果还要进一步走标准化梯度，就看 版本路线

Upgrade Signal

哪些信号说明你该从免费入口升级到正式交付

  • 需要一个客户自己的正式访问网址
  • 需要独立租户和更明确的数据 / 能力隔离
  • 需要正式账号、权限、验收和后续更新说明
  • 需要把真实脚本型能力、真实业务场景和真实审批边界接入平台
  • 需要把平台从体验入口变成可持续交付空间

Recommended Path

常见升级路径

  • 免费入口适合先体验平台与执行链路
  • 如果只是个人持续使用，先走本地脚本位轻量扩容
  • 标准团队版适合需要多人协作、正式租户和更稳定周期的场景
  • 正式交付适合需要独立网址、租户隔离和长期使用的场景
  • 标准化小团队方案和更完整的套餐能力会继续扩展
  • 常见路径是先走免费线，再按本地脚本位、团队版或企业交付逐步上探

Deployment Prep

部署评估前，通常先对齐这 3 件事

Files文件上传 / 结果下载

如果项目依赖输入文件、批次状态、Excel 结果或导出产物，先把文件链路说明清楚，再谈部署方式会更稳。

Deliverables交付物 / 文档入口

正式交付评估通常会先明确客户最终拿到什么网址、什么账号、什么说明和什么结果入口。

Checklist客户接入准备清单

已有真实脚本型能力和真实场景时，先把脚本目录、README、样例文件、依赖和风险边界准备好，会更容易完成部署判断。

Standard Delivery

标准交付内容

  • 一个独立租户
  • 一个独立访问网址
  • 一组客户专属能力，当前通常以脚本型能力为主
  • 平台账号与基础权限
  • 客户使用说明与脚本热更新说明

Standard Scope

当前默认不包含什么

  • 公开第三方能力市场
  • 客户绕过平台自行发布脚本
  • 客户自行上传任意 Vue / JS 当成正式前端页面
  • 无审批的高风险自动执行
  • 打破租户边界的跨客户能力共享

Menu Boundary

菜单定制边界

  • 客户可以通过数据库菜单挂已有页面
  • 客户可以挂外链或 iframe 类型入口
  • 如果要新增当前前端包里不存在的新页面，仍需要正式前端开发、打包和部署
  • 共享 SaaS 当前不支持让租户自行上传任意前端代码直接变成正式页面

How To Choose

按项目重点进入合适的部署路径

Customer Delivery客户交付型项目

优先单租户交付模式，再视情况补统一治理能力。

Platform Ops平台治理与运营

在租户隔离之上叠加平台治理模式，统一公告、模板、日志和台账。

Delivery Ops客户上线与后续更新

直接看客户流程 / 交付页，把标准交付包、使用路径、热更新和验收重点一次看清。

File Flow如果项目核心是文件处理和结果产物

先把上传输入文件、批次状态和结果下载的公开链路讲清楚，再决定是云端处理还是混合执行。

Delivery Kit正式交付包与说明入口

继续看交付物 / 文档入口页，把客户长期使用时真正能拿到的说明和入口一次说清。

Growth Entry免费线与分流入口

让访客先从免费线进入，再判断是继续留在个人空间、进入团队版，还是继续走企业交付路径。

CLI Bridge本地环境与内网执行

先看 CLI 指引，把 login / register / run / agent 的当前接入边界和落地路径跑通。

Ready First接入资料尚未齐备

先回到接入准备清单，而不是急着先选私有化、平台治理或本地桥接其中一种部署标签。

Current Stage当前阶段与下一步重点

先回到当前阶段说明，确认哪些是现在能做的，哪些还不该讲太满。
