Customer Flow & Delivery

客户使用流程 / 交付

Practical View交付不是把页面上线就结束，而是让客户后续还能继续安全地新增、更新和使用能力。

客户使用路径、交付边界和后续更新机制都需要清晰定义，项目才能稳定运行。

ExecGov 当前最需要先讲清楚的几件事是： 客户拿到什么、客户怎么用、平台怎么交付。 脚本持续更新流程已拆到《热更新说明》，这里主要面向标准团队版和企业交付阶段，先把“真实交付时到底怎么跑”直接展开。

Delivery Package

正式团队版或企业交付时，客户会拿到什么

  • 一个独立租户
  • 一个独立访问网址
  • 一组客户专属脚本能力
  • 平台账号与基础权限
  • 客户使用说明与客户脚本热更新说明

重点不是“给一个后台账号”，而是交付一套客户独立可运行、可更新、可继续扩展的 AI 执行空间。

Boundary

交付边界当前怎么定

  • 正式团队版和企业交付都按租户隔离，访问入口按 Host 绑定到对应租户
  • 客户只在自己的租户空间里使用自己的专属能力
  • 平台优先保证安全隔离和稳定执行
  • 标准 SaaS 场景下，共享的是前端发版节奏，不共享租户数据和权限
  • 当前不默认包含公开第三方能力市场、客户自行发版、无审批高风险自动执行

Shared SaaS Boundary

标准 SaaS 共享租户怎么理解

当前成立的部分

标准 SaaS 客户当前可以走共享租户前端。运行时仍按 Host 绑定租户 schema，所以客户看到的仍然是自己的菜单、账号、能力和数据边界。

当前不该误判的部分

这不等于每个客户都拥有一套独立前端代码副本。共享的是正式前端包与发版节奏，真正更重的专属前端、品牌化和深度定制仍然要走正式迭代或单独交付。

End To End

从需求确认到热更新生效，一条链路怎么走

01. 明确客户场景

先确认客户要解决什么业务问题、要交付哪些脚本能力、哪些需要 AI 推荐、哪些需要人工确认。

02. 开通租户和入口

平台按项目开通独立租户、独立访问网址、初始账号和对应权限边界。

03. 首批能力接入

平台把首批脚本完成审核、同步、注册和授权，确保客户登录后就能直接看到自己的能力。

04. 客户开始使用

客户通过页面手动执行，或在智能大脑中输入需求，由平台在当前租户范围内推荐能力。

05. 运行中持续反馈

客户根据执行结果、公告信息和实际业务使用情况，提出补充能力或更新脚本的需求。

06. 按热更新流程提交

客户补充脚本目录、README、所属租户、是否允许 AI 调用、是否需要人工确认等信息。

07. 平台完成治理动作

平台继续执行审核、同步、注册、版本登记、授权绑定和权限验证，不需要重新部署整个平台。

08. 当前租户内正式生效

更新后的能力只在所属租户内生效，其他租户不可见、不可执行，AI 推荐范围也不会串租户。

Menu Rules

菜单定制边界

  • 已有正式页面可以通过数据库菜单直接挂载
  • 外链和 iframe 入口也可以纳入菜单体系
  • 如果客户要的是当前前端包里没有的新页面，仍然要进入正式前端开发和重新部署流程
  • 当前共享 SaaS 不支持让租户自行上传任意 Vue / JS 成为正式产品页面

Customer Usage

客户当前怎么使用平台

1. 登录自己的网址

客户交付完成后，会拿到访问网址、登录账号和初始密码或初始化方式。登录后进入的是自己的独立租户空间。

2. 看到自己的能力，而不是源码

客户看到的是当前租户下被授权的能力列表，而不是底层脚本源码路径或其他客户的数据。

3. 选择手动执行或 AI 对话执行

客户可以直接手动执行，也可以通过智能大脑输入自然语言，让平台在当前租户和权限范围内推荐能力。

Manual Run

手动执行的基本流程

  1. 登录平台
  2. 找到当前可见的能力列表
  3. 选择目标能力
  4. 按要求填写输入信息
  5. 提交执行
  6. 查看执行结果或产物

AI Assisted

AI 对话执行的基本流程

  1. 在智能大脑页面输入需求
  2. 平台在当前租户和当前权限范围内推荐可用能力
  3. 若能力需要确认，则先人工确认
  4. 确认后平台执行对应脚本
  5. 页面返回执行结果

Notice & Result

客户还能看到什么

  • 执行结果摘要、产出内容和相关输出信息
  • 平台公告与本租户公告
  • 公告标题、来源、发布人、发布时间和正文
  • 文件上传与结果下载等当前已开放的客户侧入口

Safety Rule

使用时需要注意的边界

  • 客户只能看到自己租户下被授权的能力
  • 看不到其他客户的脚本和执行记录
  • AI 不会推荐其他客户的能力
  • 部分高风险能力需要人工确认后才会执行

Local Execution

如果脚本依赖本地环境或内网资源怎么办

这类场景通常不是把内网目录直接暴露给 Web 页面，而是通过 CLI / Agent 做桥接。平台继续负责入口、规划、权限和审计，本地环境负责执行。

Who Owns What

上线后，客户和平台各自负责什么

  • 客户负责提供脚本、README、场景说明和更新需求
  • 平台负责审核、同步、注册、授权和权限验证
  • 客户负责在自己业务里确认结果是否满足预期
  • 平台负责保证租户边界、执行安全和能力治理逻辑

FAQ

客户和售前最常问的几个问题

一开始是不是必须私有化

不一定。当前更应该先看数据边界、客户规模和执行位置，再决定是单租户交付、平台治理组合，还是混合执行路线。

热更新会不会影响其他客户

不会。脚本更新完成后，仍然只在所属租户范围内注册和授权，其他租户不会自动获得可见或可执行权限。

客户能不能自己上传脚本后直接上线

当前 1.0 默认不这样做。平台仍保留审核、注册、授权和权限验证流程，不支持绕过平台直接正式上线。

AI 会不会推荐到别的客户能力

不会。当前推荐范围受租户和权限共同约束，AI 不会跨租户推荐其他客户的脚本或能力。

本地脚本和内网资源能不能接入平台

可以，但通常不是让浏览器直接碰本地路径，而是通过 execgov-cli 和后续 Agent 路线承接本地执行桥接。

免费版和企业交付是不是两套完全无关的系统

不是。两条线当前共用同一套可信执行底座，只是入口语义、交付边界和目标用户不同。

Acceptance

客户验收重点

  • 是否能登录自己的网址
  • 是否只能看到自己的能力
  • 是否能执行自己的能力
  • 是否能查看执行结果或结果文件
  • 是否可以按流程继续更新脚本

One Sentence

一句话总结交付主线

ExecGov 交付给客户的，不只是一个登录入口，而是一套客户独立可运行、可更新、可继续扩展的 AI 执行空间；热更新也不是重部署平台，而是在当前租户范围内完成审核、注册、授权后正式生效。
