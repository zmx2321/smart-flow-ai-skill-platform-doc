Not Building

平台当前不是在做什么

不是自由聊天系统

重点不是把模型接出来，而是让真实任务在规则里被稳定执行。

不是企业后台简单裁剪版

免费版不是把复杂后台菜单直接开放，而是做独立、轻量、可增长的正式入口。

不是 demo 脚本拼装站

它要承接企业交付、平台治理、产品升级和未来生态，不是临时脚本集合。

Core Problems

平台当前在解决什么

01

企业交付问题

一个客户一个租户，一套后端能力中心支撑多个交付前端，平台超管统一治理客户、租户、模板、日志和审计。

02

AI 可信执行问题

AI 先给出计划，再等待确认；关键执行必须留痕、可审计、可回放，而不是自由越权调用脚本。

03

产品入口问题

企业交付承接收入，个人免费版承接增长入口，CLI 承接本地脚本接入与未来本地 Agent。

Free Entry Reality

免费入口的当前真实行为

公开入口已经成型

  • /experience 已经是正式免登录体验页，不再只是原型。
  • 体验页当前提供 hello_demo 与 clean_log_demo 两个公开 demo。
  • 公开路径当前是：体验页 -> /register / /login -> 正式个人版。
  • 注册成功目前通过成功提示框承接，再跳回登录页，没有单独的注册成功页。

访问统计口径要写清楚

  • 体验页展示 累计访问、今日访问、本月访问、最近一次 四张统计卡片。
  • 当前统计不是按唯一 IP 去重，而是按访问日志条数汇总。
  • 前端用浏览器会话控制，同一浏览器会话只记录一次访问。
  • 后端仍会记录 visitor_ip、user_agent 和 referer 作为公开入口日志。

Reading Order

公开文档建议阅读顺序

Step 01应用场景

先看这套平台解决哪些真实业务场景，而不是先记一堆模块名。

Step 02核心能力

再看多租户、可信执行、模板分发、公告治理和 CLI 桥接如何形成闭环。

Step 03客户流程 / 交付

直接看客户怎么用、平台怎么交付，以及脚本热更新当前如何闭环。

Step 04架构说明

理解后端、超管、免费版、交付前端和本地接入之间的关系。

Step 05CLI 指引

直接看当前已经落地的 login / register / list / run / agent 命令边界。

Step 06部署方式

看平台如何覆盖单租户交付、平台治理、免费入口和本地混合执行。

Step 07版本路线

最后看 1.0 到 4.0 的演进逻辑，以及为什么路线不是“多加几个菜单”。

Current Rules

当前核心口径

  • public 是平台治理域，不是普通租户。
  • 免费版是“我的技能库”入口，不是企业后台裁剪版。
  • 平台模板、租户脚本代理、平台自有脚本三条线必须分开。
  • 单项目先改，最后再同步到聚合壳层。

Good Fit

当前适合谁

  • 需要把已有 Python / 自动化脚本接成 AI 可调度能力的团队
  • 需要一套可继续扩租户、扩模板、扩技能的交付底座
  • 需要同时运营企业项目和个人免费入口的独立开发者 / 小团队
